## Project Overview

We are building a static site using the Swift [Publish](https://github.com/johnsundell/publish) framework. The site is defined as a Swift executable that generates HTML from Markdown content.

## Build & Run

```bash
# Generate + commit + push (Cloudflare Pages serves the committed Output/)
make publish

# Generate Output/ only, no commit/push
make generate          # equivalent: publish generate

# Generate and serve locally at http://localhost:8000
make preview           # equivalent: publish run

# Run the Arcade Hall JavaScript and generated-route tests
make test-games

# Build the Swift package / clean artifacts
swift build
make clean             # equivalent: swift package clean
```

> [!IMPORTANT]
> Do not manually edit anything inside the `Output/` directory ŌĆö it is strictly generated by Swift for our Cloudflare Pages deployment.

## Architecture

- **`Package.swift`** ŌĆö SPM manifest. Single executable target `KinNoKiLabsSite` depending on `Publish` (and transitively on Plot, Ink, Files, ShellOut, Sweep).
- **`Sources/KinNoKiLabsSite/main.swift`** ŌĆö All core website structure and theme logic is handled here. Defines the `KinNoKiLabsSite` struct conforming to `Website`, which configures site metadata (URL, name, sections) and the publish pipeline. Currently uses the custom `.kinNoKi` theme.
- **`Sources/KinNoKiLabsSite/Theme/KinNoKiTheme.swift`** ŌĆö Custom Plot-based `HTMLFactory` theme providing a sticky navigation bar and modern layout. The theme generates full HTML documents with shared `<header>`, `<main>`, and `<footer>` across all page types (index, sections, items, pages, tags).
- **`Content/`** ŌĆö Markdown source files. Top-level `.md` files become pages (e.g., `about.md` ŌåÆ `/about`); subdirectories become sections (e.g., `Content/posts/` ŌåÆ `SectionID.posts`).
- **`Resources/`** ŌĆö Static assets (images, CSS, JS) copied verbatim to the output root. `styles.css` defines all site styling including the sticky nav, typography, and light/dark mode support.
- **Arcade Hall** ŌĆö `/games`, `/games/sudoku`, `/games/crossword`, and `/games/word-search` are generated from the matching files under `Content/`. The reusable puzzle engines and shared helpers live in `Resources/games/core.js`, `controller-common.js`, and the game modules; DOM rendering and interaction adapters live in the matching `*-ui.js` controllers. Browser state, including preferences and puzzle progress, stays local to that browser and is never uploaded.
- **`Tests/games/`** ŌĆö Node tests cover the pure game engines, DOM controllers, accessibility behavior, and generated routes. Run them with `make test-games` after game changes; generate the site before running route assertions directly.

### Publish Conventions

- **Site config** lives in the `Website` conformance (`url`, `name`, `description`, `language`, sections via `SectionID` enum).
- **Sections** are declared as cases of `SectionID: String, WebsiteSectionID`. Each section maps to a subdirectory in `Content/`.
- **Item metadata** can be added via frontmatter in content files (YAML between `---` delimiters) and parsed into `ItemMetadata`.
- **Themes** wrap the site in HTML structure. Custom themes are created by conforming to `Theme` with an `HTMLFactory` (see `Theme/KinNoKiTheme.swift`). Note: `resourcePaths` on `Theme` is only for resources bundled inside a theme package ŌĆö project-level assets in `Resources/` are copied automatically by Publish.
- **Publishing steps** are customizable via `publish(using:)` instead of the default `.publish()` call (e.g., to add plugins, custom deployment, or pipeline steps).

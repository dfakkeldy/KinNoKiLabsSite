# KinNoKi App Page Mirror Truth Sync Design

Date: 2026-07-05

## Summary

Pass 1 updates the KinNoKi Labs umbrella site so its app pages faithfully mirror
the public GitHub Pages homepages for the app portfolio. The goal is truth sync,
not repositioning: make the public umbrella site stop lagging the product
surfaces while preserving the current Publish theme and site structure.

Press, broader brand storytelling, and homepage repositioning are intentionally
out of scope for this pass.

## Goals

- Add Routey to the KinNoKi app portfolio.
- Rename public umbrella references from Visual Timer to Turn Timer.
- Refresh all KinNoKi app pages from their corresponding GitHub Pages homepages.
- Keep app pages accurate without importing each app site's custom layout,
  styles, scripts, decorative HTML, or data URI artwork.
- Update the homepage, support page, and privacy page so their app lists match
  the five-app portfolio.

## Non-Goals

- Do not create a `/press` page in pass 1.
- Do not redesign the KinNoKi homepage or reposition the studio story.
- Do not change the shared Publish theme except for small fixes required by the
  content sync.
- Do not claim App Store availability, public launch status, social handles, or
  press assets unless already stated on the source app site or separately
  verified during implementation.
- Do not copy generated `Output/` files by hand.

headlines, status language, features, and links, but adapt them into simple
Markdown that fits the KinNoKi site.

## Page Structure

Each mirrored app page should use a consistent Markdown shape:

1. H1 with the app name or published homepage headline.
2. Short positioning paragraph copied or closely adapted from the source site.
3. Major sections from the source homepage, preserving section order where it
   makes sense in Markdown.
4. Feature lists using headings and bullets rather than custom cards.
5. Status notes only when present in the source site or verified during the
   implementation pass.
6. Footer links to the source GitHub Pages site, repository, and support route
   where available.

For Turn Timer, keep `/apps/visualtimer` for now and label it as Turn Timer. A
future pass can decide whether to add a `/apps/turntimer` alias or migrate URLs.

## Architecture

The implementation stays inside the existing Swift Publish site:

- Source content lives in `Content/`.
- App pages live in `Content/apps/`.
- Shared layout remains in `Sources/KinNoKiLabsSite/Theme/KinNoKiTheme.swift`.
- Styling remains in `Resources/styles.css` unless a content-driven bug appears.
- Generated output remains under `Output/` and must be regenerated by Publish,
  not edited manually.

No new dependency or third-party framework is needed.

## Data Flow

1. Read each live GitHub Pages homepage and, when useful, its local source file.
2. Convert the app homepage content into KinNoKi markdown.
3. Add or update the `Content/apps/*.md` pages.
4. Update the homepage, support page, and privacy page.
5. Run site generation so `Output/` reflects the source changes.
6. Verify the generated pages locally.
7. Commit source and generated output together because Cloudflare Pages serves
   the committed `Output/` directory in this repo.

## Error Handling And Truth Rules

- If a source app homepage contains a stale or unverifiable claim, preserve only
  the claim as written on that source page or omit it; do not upgrade it.
- If a source app homepage has complex visual-only content, summarize the
  actual user-facing claim rather than copying markup.
- If a live source page fails to load during implementation, fall back to the
  local published source file only after noting that the live page could not be
  refreshed.
- If Routey or another app lacks a local help page, link to GitHub or the app
  page instead of fabricating support copy.
- Keep private route data, press notes, credentials, and unpublished assets out
  of the site.

## Verification

Implementation should verify:

- `publish generate` or `make generate` succeeds.
- Generated `Output/apps/routey/index.html` exists.
- Generated pages use `Turn Timer`, not `Visual Timer`, except where referring
  to the historical repository name or URL.
- `Output/index.html`, `Output/support/index.html`, and
  `Output/privacy/index.html` mention all five apps.
- Live-source links resolve before the implementation is committed when network
  access is available.
- `git status --short --branch` is checked before finishing.

There are no GitHub Actions workflows in this repo, so local generation and a
manual preview check are the primary pass-1 verification gates.
